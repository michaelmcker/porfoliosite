Selected workflow · Publishing system

Presentation publishing

Moves approved branded HTML to a managed, prospect-ready URL without a manual deployment or CMS handoff.

01 · Purpose

Why it exists

Custom HTML presentations can be more expressive than a standard slide deck, but deployment and CMS setup create a gap between the approved artifact and the link a sales team can actually share. This workflow connects those layers while keeping the approved presentation intact.

02 · Sequence

How it works

The same six stages appear in the visual and in this public-safe operating sequence.

  1. 01

    Branded HTML presentation

    Start with a self-contained, approved HTML artifact plus its intended name and destination.

  2. 02

    Vercel API deployment

    Publish the presentation into the corporate Vercel project as its own hosted artifact.

  3. 03

    Name and destination validation

    Check the proposed name and URL against existing presentations before making a CMS change.

  4. 04

    Webflow CMS iframe field

    Create or update the CMS record and write the shared iframe embed field through the Webflow CMS API.

  5. 05

    Full-screen presentation template

    The managed CMS page renders the Vercel-hosted artifact edge to edge.

  6. 06

    Published URL verification

    Open and verify both publishing layers, then return the final prospect-ready URL.

03 · Sources

Tools and data sources

The workflow is deliberately narrow: it handles publishing infrastructure, not presentation authorship.

  • self-contained HTML
  • publishing skill
  • Vercel API
  • Webflow CMS API
  • existing-presentation lookup
  • iframe embed field
  • browser verification

Output

What ships

A Vercel-hosted artifact, a managed Webflow page, and a final prospect URL ready to share.

Review

What stays human

Final content and design approval, plus confirming the intended name and destination before publication.

04 · Boundary

Public-safe proof and constraints

The deterministic diagram and published-page structure show the two-layer system without exposing credentials, tokens, internal identifiers, or private client content. The workflow does not invent presentation content, silently change an approved artifact, or claim a publish succeeded before both URLs are checked.
